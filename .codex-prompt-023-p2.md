# PRD-023 Phase 2: Event Discovery Pipeline

## Context
Phase 1 is merged and live. The database tables exist (`tracked_cities`, `city_events`, `tracked_venues`, `city_sources`, `event_pipeline_diary`, `event_source_reports`, `event_feedback`, `user_event_interests`). 15 cities are seeded. The city pages UI exists at `/cities` and `/cities/[slug]`. But the pages are empty — no events have been discovered yet.

Your job: build the event discovery pipeline that populates `city_events` with real, high-quality events.

## What to Build

### 1. Discovery Script (`scripts/discover-events.ts`)

A TypeScript script (runnable via `npx tsx scripts/discover-events.ts`) that:

1. **Reads tracked cities** from Supabase (`tracked_cities` table)
2. **For each city**, discovers events using multiple strategies:
   a. **RSS feeds** from `city_sources` table (if any exist for this city)
   b. **Web search** via Brave Search API for "[city] events [month] [year]", "[city] exhibitions [month] [year]", genre-specific searches
   c. **Structured API queries** to known aggregators (Eventbrite, Bandsintown/Songkick if available)
3. **Deduplicates** events (same title + same venue + overlapping dates = duplicate)
4. **Scores quality** (0-100) using an AI quality gate:
   - "Would a well-traveled person be disappointed to miss this?"
   - Venue prestige (from `tracked_venues.tier` if venue is tracked)
   - Artist/performer recognition (search presence, Wikipedia article)
   - Event scale (festival > single show > recurring)
5. **Only inserts events scoring 60+** into `city_events`
6. **Writes a pipeline diary** entry to `event_pipeline_diary` with:
   - What sources were checked
   - How many events found, how many new vs duplicates
   - What worked, what failed
   - What to try tomorrow (AI-generated next-day plan)
7. **Reads yesterday's diary** before starting and adapts strategy accordingly
8. **Tracks source health** in `event_source_reports` (success/error/events found per source)

### 2. Source Discovery Script (`scripts/discover-sources.ts`)

A script that bootstraps sources for a city (Day 0 onboarding):

1. Takes a city slug as argument
2. Searches for RSS feeds: "[city] events RSS", "[city] tourism board", "[city] what's on calendar"
3. Checks known URL patterns: `visit[city].com`, `[city]tourism.com`, `timeout.com/[city]`
4. Auto-discovers RSS/Atom feeds via `<link rel="alternate">` on found pages
5. Adds discovered feeds to `city_sources` with status 'candidate'
6. Adds global aggregators (Bandsintown, Songkick, Eventbrite, Music Festival Wizard) for every city

### 3. API Route for Manual Event Refresh (`POST /api/v1/events/refresh`)

Admin-only endpoint that triggers discovery for a specific city:
```
POST /api/v1/events/refresh
{ "citySlug": "paris" }
```

### 4. Cron Configuration

The pipeline should be designed to run as a daily cron job. Add a commented example showing how to call it. The script should:
- Accept `--city <slug>` to run for a single city (for testing)
- Accept `--all` to run for all tracked cities
- Accept `--dry-run` to show what would be discovered without writing to DB
- Log clearly to stdout

## Technical Constraints

- **Stack:** TypeScript, Next.js App Router, Supabase (use `@supabase/supabase-js`)
- **AI calls:** Use OpenAI SDK (`openai` package) with `gpt-4o` for quality scoring and diary writing. The OPENAI_API_KEY is in the environment (not .env.local). Read it from `process.env.OPENAI_API_KEY`. If not set, fall back to Google Gemini via `@google/generative-ai` package using GEMINI_API_KEY from environment.
- **Web search:** Use Brave Search API. The BRAVE_API_KEY is in `.env.local`. Endpoint: `https://api.search.brave.com/res/v1/web/search`
- **RSS parsing:** Use `rss-parser` npm package (add to dependencies)
- **Database:** Use Supabase service client for the pipeline (this runs as a backend script, not user-facing). Import from `@/lib/supabase/service`. Call `createSecretClient()`.
- **Do NOT use `createSecretClient()` in any API route or user-facing code.** Only in the pipeline scripts.
- **No real names, addresses, or PII in code comments, test data, or logs.**
- **RSS feeds that fail 10 consecutive times should be marked dormant** and skipped.

## Quality Scoring Guidelines

The AI quality gate prompt should evaluate:
- **Venue significance** (world-famous museum > local bar)
- **Artist/performer significance** (Pulitzer Prize winner > unknown local act)
- **Event uniqueness** (once-in-a-lifetime retrospective > weekly open mic)
- **Cultural importance** (major festival > corporate meetup)
- **Duration/scale** (multi-day festival > 1-hour talk)

Score thresholds:
- 80-100: Major (Basquiat at FLV, Firenze Rocks, Umbria Jazz)
- 60-79: Medium (notable gallery show, mid-tier concert, food festival)
- Below 60: Don't insert

Tier mapping:
- 80+: 'major'
- 60-79: 'medium'

## Deduplication Strategy

Events are duplicates if:
- Same city AND
- Title similarity > 80% (use Levenshtein or normalized comparison) AND
- Date ranges overlap AND
- (Same venue OR venue is null)

When a duplicate is found, keep the one with more information (longer description, has image, has booking URL).

## Important Notes

- The Brave Search API has rate limits. Add 1-second delays between requests.
- When searching for events, use the local language for non-English cities (French for Paris, Italian for Florence, Japanese for Tokyo).
- The pipeline should be idempotent — running it twice should not create duplicates.
- Set `last_verified_at` to now() for newly discovered events.
- Set `expires_at` to `end_date + 1 day` (or `start_date + 1 day` if no end_date).
- Update `tracked_cities.last_refreshed_at` after a successful run.
