## Problem

Ian is flying Miami → NYC → Austin → Portland → Charleston → Milan. He asked Jacques to manually cross-reference weather forecasts for each city with his travel dates to figure out what to pack. This required six separate weather API calls, city-by-city analysis, and a hand-written summary combining everything into packing advice.

Every multi-city traveler has this problem. A March trip that spans Miami (82°F, humid) and NYC (38°F, windy) means you need swimwear AND a winter coat in the same suitcase. Nobody wants to check six weather apps and mentally merge the results.

This should be a first-class feature. When you open your trip, you should see weather for every stop and an AI-generated packing list that accounts for the full temperature swing.

## What This Is

**Weather forecasts per destination + AI packing suggestions for the whole trip.**

When a user views a trip in UB Trippin:
1. Each city shows a weather card with high/low temps, precipitation chance, and conditions for the dates they'll be there
2. A single consolidated packing list accounts for ALL destinations — not just one city at a time
3. A temperature range visualization shows the min-to-max swing across the entire trip

## Design Decisions

### Weather Data Source → Open-Meteo API

**Open-Meteo** is free, requires no API key, and provides a 16-day forecast window. This is the right choice for v1:

- **Forecast endpoint:** `https://api.open-meteo.com/v1/forecast` — used for trips within the 16-day window
- **Geocoding endpoint:** `https://geocoding-api.open-meteo.com/v1/search` — resolve city names to lat/lng

> **Evaluated:** OpenWeatherMap (used in Leone weather app, API key available). OWM One Call 3.0 only provides 8-day forecasts vs Open-Meteo's 16 days. The longer forecast window matters more for a travel app. OWM also has per-call costs at scale ($0.0015/call after free tier). Open-Meteo is the better fit. WeatherLink (also in Leone) is a physical station API — not applicable for multi-city forecasts.

Parameters we request per city:
```
latitude, longitude
daily: temperature_2m_max, temperature_2m_min, precipitation_sum,
       precipitation_probability_max, weathercode, wind_speed_10m_max
temperature_unit: fahrenheit (default, user-configurable)
timezone: auto
```

### When to Show Weather → 16-Day Window Rule

- **Trip > 16 days away:** Show nothing. No placeholder, no averages. Weather appears when we have real forecast data.
- **Trip within 16 days:** Show live forecast data with daily granularity
- **Trip in progress:** Show current conditions + remaining days forecast
- **Trip completed:** Show nothing (or "Trip complete" state)

Weather data refreshes:
- **Auto-refresh:** Every 6 hours for trips within the 16-day window
- **Manual refresh:** User can pull-to-refresh or click a refresh button
- **On trip view:** Fetch fresh data if last fetch > 1 hour ago

### Location Resolution → Extract from Trip Items

Use existing `start_location` and `end_location` fields on `trip_items` to build the city list. The extraction pipeline already populates these (e.g., "Miami, FL", "JFK Airport, New York, NY").

**City extraction logic:**
1. Collect all unique locations from trip items, **prioritizing hotel locations** — if a hotel exists for a city, use that location (it's where the traveler actually is). Fall back to airport/flight locations only if there's no hotel.
2. Normalize to city-level (strip airport codes, "Airport" suffix, hotel names, etc.)
3. Deduplicate (JFK and LGA both → "New York")
4. Order chronologically by the earliest `start_date` for that city
5. Determine date ranges per city from the trip items

**Edge case handling:**
- **Layovers (< 6 hours in a city):** Exclude. No need for weather on short layovers.
- **Overnight layovers:** Include. If they're sleeping there, weather matters.
- **No location data:** Show "Add locations to your trip items to see weather forecasts"
- **One-way trips / no return:** Show weather for all cities with known dates
- **Duplicate cities (depart and return through same hub):** Show once, with the combined date range

### Packing List Generation → AI-Powered

Use the same AI extraction pipeline pattern (OpenAI/Anthropic) we use for email parsing:

**Input to the model:**
```json
{
  "trip_title": "Spring 2026 Tour",
  "destinations": [
    {
      "city": "Miami, FL",
      "dates": "Mar 15-18",
      "weather": { "high": 84, "low": 72, "precip_chance": 20, "conditions": "Partly cloudy" }
    },
    {
      "city": "New York, NY",
      "dates": "Mar 18-21",
      "weather": { "high": 45, "low": 32, "precip_chance": 60, "conditions": "Rain/snow mix" }
    }
  ],
  "temp_range": { "min": 32, "max": 84, "unit": "°F" },
  "traveler_count": 1
}
```

**Output structure:**
```json
{
  "essentials": ["Passport", "Phone charger", "Medications"],
  "clothing": [
    { "item": "Light layers for Miami heat (82-85°F)", "reason": "miami" },
    { "item": "Heavy winter coat for NYC (32-45°F)", "reason": "nyc" },
    { "item": "Rain jacket or umbrella", "reason": "portland, nyc" }
  ],
  "footwear": ["Comfortable walking shoes", "Sandals for Miami"],
  "accessories": ["Sunglasses", "Warm hat and gloves for NYC"],
  "tip": "You're spanning a 50°F temperature range — layer strategy is key. Pack a compressible down jacket that works for NYC cold and Portland drizzle."
}
```

### Caching Strategy → Supabase Table

Weather data changes, but not every minute. Cache in a new table to avoid hammering the API:

```sql
create table public.trip_weather_cache (
  id uuid primary key default gen_random_uuid(),
  trip_id uuid not null references public.trips(id) on delete cascade,
  user_id uuid not null references public.profiles(id) on delete cascade,
  city text not null,
  latitude numeric(9,6) not null,
  longitude numeric(9,6) not null,
  date_start date not null,
  date_end date not null,
  forecast_json jsonb not null,        -- daily forecast array from Open-Meteo
  source text not null default 'forecast',  -- 'forecast' only (no climate averages)
  packing_json jsonb,                  -- AI-generated packing list (trip-level, stored on first city row)
  fetched_at timestamptz not null default now(),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

-- RLS: users can only see their own weather data
alter table public.trip_weather_cache enable row level security;

create policy "Users see own weather data"
  on public.trip_weather_cache for select
  using (user_id = auth.uid());

create policy "Users insert own weather data"
  on public.trip_weather_cache for insert
  with check (user_id = auth.uid());

create policy "Users update own weather data"
  on public.trip_weather_cache for update
  using (user_id = auth.uid());

create policy "Users delete own weather data"
  on public.trip_weather_cache for delete
  using (user_id = auth.uid());

-- Index for quick lookups
create index idx_weather_cache_trip on public.trip_weather_cache(trip_id);
create index idx_weather_cache_user on public.trip_weather_cache(user_id);
```

**Cache invalidation:**
- Stale after 6 hours for trips within forecast window
- No climate averages — only real forecast data cached
- Bust on trip item changes (add/edit/delete items that affect cities or dates)

## API Endpoints

### `GET /api/v1/trips/:id/weather`

Returns weather data for all destinations in a trip.

**Response:**
```json
{
  "trip_id": "uuid",
  "temp_range": { "min": 32, "max": 84, "unit": "°F" },
  "destinations": [
    {
      "city": "Miami, FL",
      "latitude": 25.7617,
      "longitude": -80.1918,
      "dates": { "start": "2026-03-15", "end": "2026-03-18" },
      "source": "forecast",
      "daily": [
        {
          "date": "2026-03-15",
          "temp_high": 84,
          "temp_low": 72,
          "precipitation_mm": 0.5,
          "precipitation_probability": 20,
          "weather_code": 2,
          "weather_description": "Partly cloudy",
          "wind_speed_max_mph": 12
        }
      ]
    }
  ],
  "packing": {
    "essentials": [...],
    "clothing": [...],
    "footwear": [...],
    "accessories": [...],
    "tip": "..."
  },
  "fetched_at": "2026-03-06T10:00:00Z",
  "is_stale": false
}
```

**Query params:**
- `?refresh=true` — force fresh fetch, bypass cache
- `?unit=celsius` — override temperature unit (default: user preference or fahrenheit)

**Auth:** API key (same as other v1 endpoints)

### `POST /api/v1/trips/:id/weather/refresh`

Force-refresh weather data and regenerate packing list. Rate-limited to 1 per minute per trip.

## Edge Cases

| Scenario | Behavior |
|----------|----------|
| Trip > 16 days away | Show nothing. Weather section hidden until forecast data is available. |
| Layover < 6 hours | Skip city, don't show weather |
| Overnight layover (6+ hours) | Include city with weather |
| No location data on trip items | Show empty state with guidance |
| City name ambiguous (Portland, OR vs Portland, ME) | Use state/country from location string; fall back to asking geocoding API and picking highest-population match |
| Trip spans > 16 days total | Show forecast for days within window only. No data for dates beyond forecast range. |
| Open-Meteo API down | Show cached data with "Last updated X ago" warning; degrade gracefully |
| Single-city trip | Still show weather card + packing suggestions (simpler list) |
| International cities | Open-Meteo works globally; geocoding handles international names |
| User changes trip items | Invalidate weather cache, re-extract cities on next view |
