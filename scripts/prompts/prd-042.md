## Part 1: Rich FlightAware Data

### 1A. Extend the flight status data model

Currently `src/lib/flight-status.ts` fetches basic status. Extend `getFlightStatus()` to also extract and return:

```typescript
// Add these fields to the status response stored in flight_status_cache
{
  // Aircraft
  aircraft_type: string | null,      // e.g. "A320neo" — from FA field "aircraft_type"
  tail_number: string | null,        // e.g. "N912NK" — from FA field "registration" 
  
  // Inbound aircraft
  inbound_fa_flight_id: string | null,  // FA flight ID of inbound aircraft
  inbound_origin: string | null,        // e.g. "LAX" — origin of inbound flight
  inbound_ident: string | null,         // e.g. "NK402" — flight number of inbound
  inbound_estimated_in: string | null,  // ETA of inbound aircraft
  
  // Codeshare
  operator: string | null,           // actual operating carrier if codeshare
  operator_iata: string | null,      // IATA code of actual operator
  codeshares: string[] | null,       // list of codeshare flight numbers
  
  // Arrival details  
  baggage_claim: string | null,      // carousel number, available after landing
  arrival_gate: string | null,       // arrival gate (may differ from departure gate)
  arrival_terminal: string | null,   // arrival terminal
  
  // Departure details (already partially have these)
  departure_gate: string | null,     // gate at departure airport  
  departure_terminal: string | null, // terminal at departure airport
  
  // Actual times
  actual_off: string | null,         // actual wheels-off (takeoff)
  actual_on: string | null,          // actual wheels-on (landing)
  actual_out: string | null,         // actual gate departure
  actual_in: string | null,          // actual gate arrival
}
```

These fields are ALL available in the FlightAware AeroAPI v4 flights endpoint response that we already call. We're just not extracting them. Look at the existing `getFlightStatus()` function — it fetches `flights/{ident}` and takes `flights[0]`. Extract the additional fields from that same response object.

For **inbound aircraft**: FA returns `inbound_fa_flight_id` on the flight object. If present, make ONE additional API call to `flights/{inbound_fa_flight_id}` to get the inbound flight's origin and ETA. Cache this — don't call it every refresh.

### 1B. Extend the flight_status_cache table

Add columns to `supabase/migrations/` for the new fields. The cache table is `flight_status_cache`. Add nullable columns for all the new fields listed above. Create a new migration file.

### 1C. Update the status API endpoints

Update `src/app/api/v1/items/[id]/status/route.ts` and `src/app/api/v1/items/[id]/status/refresh/route.ts` to return the new fields in the response.

### 1D. Update the live flight page API

Update `src/app/api/v1/flights/[ident]/[date]/live/route.ts` to also extract and return:
- aircraft_type, tail_number (registration)
- operator, codeshares
- baggage_claim

- actual_off, actual_on, actual_out, actual_in
- inbound flight info (if available, with a separate cached FA call)

The live route already extracts some of these (aircraft_type, gates, terminals). Ensure ALL of the above are included.

## Part 2: Flight Card Redesign

### Design Spec (from Gemini + Ian's decisions)

Create a new component `src/components/trips/flight-item-card.tsx` that replaces the flight rendering in `TripItemCard`.

**In `trip-item-card.tsx`:** When `item.kind === 'flight'`, render `<FlightItemCard>` instead of the generic card. Pass all necessary props. Keep the generic card for all other item kinds unchanged.

### Card Structure

#### Status Banner (always visible, top of card)
Full-width colored bar at the very top of the card. Rounded top corners (`rounded-t-2xl`).

Colors:
- `on_time` / `scheduled`: `bg-indigo-600 text-white` — "On Time"
- `boarding`: `bg-indigo-600 text-white` — "Boarding"  
- `delayed`: `bg-amber-500 text-white` — "Delayed · {minutes} min"
- `en_route`: `bg-blue-600 text-white` — "In Flight · ETA {time}"
- `landed`: `bg-green-600 text-white` — "Landed at {time}"
- `cancelled`: `bg-red-600 text-white` — "Cancelled"
- `diverted`: `bg-red-600 text-white` — "Diverted to {airport}"
- `unknown` / no status / outside 48h window: NO banner shown

The banner only shows when the flight is within 48 hours of departure OR in progress OR recently landed. Use the existing `isWithin48Hours` logic.

#### Collapsed View (always visible, below banner)

**Header row:**
- Airline logo (existing `getProviderLogoUrl`) + airline name (smaller, gray)
- Flight number — **links to live flight page** (`/flights/{ident}/{date}`). Use existing `buildFlightPageUrl`. Style as a link: `text-indigo-600 hover:underline font-bold text-lg`
- Confirmation code (right-aligned, mono font)

**Journey section:**
- NO flight path visual (Ian's decision — save vertical space)
- Departure block: airport code (text-2xl bold), city name, terminal + gate, time
- Arrow (→) between departure and arrival
- Arrival block: airport code (text-2xl bold), city name, terminal + gate, time
- When landed and baggage_claim available: show "Baggage Claim {number}" in green bold instead of arrival gate

**Time display logic:**
- Normal: show scheduled time, label "Scheduled"
- Delayed: show estimated time in red bold, scheduled time with line-through next to it, label "Estimated"
- Landed with actual time: show actual time in green bold, label "Actual"
- Gate/terminal should update from live FA data (use the `onStatusUpdate` callback pattern already in place)

**Keep from existing card:**
- Traveler names
- Loyalty chip warnings
- Needs review badge
- Action buttons (share, edit, move, delete menu)
- metaChips prop

#### Expanded View (on "Show Details" click)

Two-column grid on desktop (`md:grid-cols-2`), single column on mobile.

**Column 1 — Aircraft & Performance:**
- Aircraft type (e.g., "Airbus A320neo")
- Registration / tail number (e.g., "N912NK")  
- On-time performance percentage (if available from route_statistics — defer this to a follow-up if the endpoint is complex)
- Codeshare info if applicable ("Operated by {operator}")

**Column 2 — Operations:**
- Inbound flight info ("Your plane is arriving from {city} as {flight_number}")
- Link to inbound flight's live page if we have the ident
- Departure/arrival weather (from our weather system if available — can be a follow-up)

**Full-width expand button** at the bottom: "Show Details" / "Show Less" with rotating chevron. Style: `text-indigo-600 hover:bg-indigo-50 rounded-lg w-full py-2 text-sm font-semibold`

#### Cancelled state
When cancelled: entire card gets `opacity-60`. Banner is red. All details still accessible.
